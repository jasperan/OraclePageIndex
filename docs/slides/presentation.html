01 — The Problem

Similarity isn't relevance.

Vector RAG chunks text, embeds it, then returns whatever is closest in embedding space. Search for "Apple's supply chain risks" and you might get paragraphs about fruit logistics.

What we actually need: reasoning over document structure. Follow the table of contents. Cross-reference entities. Traverse relationships. That's what knowledge graphs do.

02 — The Solution

Documents become knowledge graphs.

OraclePageIndex parses PDFs into Oracle SQL Property Graphs. Entities, sections, and relationships stored as vertices and edges, queryable with standard SQL.

What You Get

  • Graph traversal over named relationships
  • Multi-hop reasoning via SQL/PGQ recursive paths
  • Explainable retrieval with traceable graph paths
  • Fully local: Ollama + Oracle Free, no API keys

08 — SQL/PGQ

The Power of Oracle Property Graphs

Every query is standard SQL using GRAPH_TABLE with MATCH patterns. No proprietary graph language. Just SQL.

SELECT e2.name, e2.entity_type, r.relationship
FROM GRAPH_TABLE (doc_knowledge_graph
  MATCH (e1 IS entity WHERE e1.name = 'Apple Inc.')
      -[r IS related_to]-> (e2 IS entity)
  COLUMNS (e2.name, e2.entity_type, r.relationship)
);
08 — SQL/PGQ (cont.)

Multi-Hop Path Queries

Find how two entities connect through intermediate nodes. The ->+ syntax traverses to any depth in a single query.

-- Find entity path: source -> intermediate -> target
SELECT e1.name, r1.relationship, mid.name,
       r2.relationship, e2.name
FROM GRAPH_TABLE (doc_knowledge_graph
  MATCH (e1 IS entity WHERE e1.name = :source)
    -[r1 IS related_to]-> (mid IS entity)
    -[r2 IS related_to]-> (e2 IS entity WHERE e2.name = :target)
  COLUMNS (e1.name, r1.relationship, mid.name,
           r2.relationship, e2.name)
);
08 — SQL/PGQ (cont.)

Recursive Hierarchy

-- ALL descendants of a section
SELECT child.title, child.summary
FROM GRAPH_TABLE (
  doc_knowledge_graph
  MATCH (parent IS section
    WHERE parent.title = 'Intro')
    -[h IS parent_of]->+ (child IS section)
  COLUMNS (child.title,
           child.summary)
);

Why This Matters

  • The ->+ operator is SQL/PGQ's recursive path expression
  • Traverses hierarchy to any depth in one query
  • No recursive CTEs needed
  • ISO SQL:2023 standard

11 — Entity Resolution

Vector-Assisted Dedup

Different documents call the same entity by different names. Ollama embeddings + Oracle VECTOR_DISTANCE find candidates, then LLM confirms merges.

entity_resolution:
  enabled: true
  embedding_model: "nomic-embed-text"
  similarity_threshold: 0.3
  auto_confirm_threshold: 0.15

Vectors serve the graph here. They're used strictly for entity name disambiguation, not for retrieval.

12 — Enrichment Agent

Automated Gap-Filling

A post-indexing pass finds isolated entities and co-occurring pairs that share sections but aren't linked. For each candidate, a focused LLM prompt examines shared text and discovers relationships.

# Full enrichment pass
python run.py enrich

# Preview candidates without writing
python run.py enrich --dry-run

# Limit scope
python run.py enrich --max-candidates 50

Enriched edges tagged with edge_source='ENRICHMENT' + confidence score.

13 — Temporal Versioning

Track Changes Across Versions

Index multiple versions of the same document. The system computes diffs: which entities APPEARED, DISAPPEARED, or remained STABLE.

python run.py index report-2023.pdf \
  --doc-group annual-report --doc-version 1

python run.py index report-2024.pdf \
  --doc-group annual-report --doc-version 2
APPEARED DISAPPEARED MODIFIED STABLE
